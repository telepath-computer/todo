# v0.2 implementation sequence

Companion to `task-status-and-notes-preamble.md` (the v0.2 spec). That doc says **what** changes; this one says **in what order** to land the changes, **what tests** each step adds, and **what acceptance bullets** each step satisfies.

## Sequencing rationale

- **Due-syntax change goes first.** Smallest, most independent change; exercises the "back-compat read, canonical write" pattern in a tiny scope before we reuse it for `## Tasks` → `## Available`. Frees `@` for contexts cleanly.
- **Lane structure (incl. the `## Tasks` → `## Available` rename) is one step, not two.** A pure rename has no user-visible behavior to test; folding it into the lane work avoids a wasted TDD cycle.
- **Contexts come after** lanes so the parser is already comfortable with multiple sections.
- **Preamble + Completed** are independent of the lane work; sequence them after the lane structure stabilises.
- **`td list` and rendering polish come last** — they consume features built in earlier steps.

Each step is a self-contained TDD increment:

1. Write tests (e2e + integration) that map to the spec's acceptance bullets for that step.
2. Implement until tests pass.
3. Update `docs/spec.md` and `docs/cli.md` to reflect the new behavior.
4. Run the full suite. Zero regressions in earlier steps.
5. Commit.

## The seven steps

### Step 1 — Due syntax `@YYYY-MM-DD` → `!YYYY-MM-DD`

**Scope.** Canonical emit becomes `!YYYY-MM-DD`. Reader accepts both `@YYYY-MM-DD` (legacy/back-compat) and `!YYYY-MM-DD`. Writer always emits `!`. Renderer shows `!YYYY-MM-DD` with the existing red/yellow/dim color rules.

This is the smallest end-to-end change in v0.2. It's also where we pin down the back-compat read pattern (`reader accepts both, writer emits canonical`) we'll reuse for `## Tasks` → `## Available` in step 2.

**Tests.**
- Integration: parse `!2026-05-01` as due; parse `@2026-05-01` as due (back-compat); round-trip a file with legacy `@` re-emits with `!`; due-color rules still apply.
- e2e: `td tasks add --due 2026-05-01` writes `- [ ] ... !2026-05-01` to disk; existing `@`-due files still produce correct `td tasks list` output and survive a complete/uncomplete round-trip.

**Acceptance.** "Due dates added via `--due 2026-05-01` serialize as `!2026-05-01`; legacy `@2026-05-01` in existing files still parses as due."

**Files.** `src/core/project.ts` (parser, serializer), `src/core/tasks.ts`, `src/views/task.ts`, `tests/project.test.ts`, `tests/tasks.test.ts`, `tests/cli.e2e.test.ts`, `docs/spec.md`.

---

### Step 2 — Three-lane structure (incl. `## Tasks` → `## Available` rename)

**Scope.** Introduce `## Waiting` and `## Deferred` as siblings of `## Available` (which replaces `## Tasks`). Reader accepts legacy `## Tasks` files (treats them as `## Available`); writer always emits `## Available`. Item gains `lane: 'available' | 'waiting' | 'deferred'`. Refs become flat across all three sections in document order.

CLI:
- `td tasks add --waiting` / `--deferred` (default `--available`). Mutually exclusive.
- `td tasks edit <ref> --waiting` / `--deferred` / `--available` (moves between lanes; standard shift-note).
- `td tasks list` default = available + waiting. `--available` / `--waiting` / `--deferred` for single-lane. `--all` for all three.

Sections: Available always emitted; Waiting + Deferred emitted only when non-empty.

**Tests.**
- Integration: parse round-trip with all three sections; lane field; ref numbering across sections; lane changes via `editTask`; section omission when empty; ref shift on lane move; legacy `## Tasks` file reads as `## Available` and re-emits with new heading.
- e2e: add into each lane, edit between lanes, list with each flag, list default = avail+wait, `--all` returns all three. Empty `## Waiting` / `## Deferred` not emitted; section appears when first item added; section disappears when last item moves out. Existing CLI tests (add/list/complete/edit/remove) updated to assert `## Available` and still pass.

**Acceptance.** Spec bullets: `tasks add --waiting`, `tasks edit <ref> --deferred`, `tasks list` default + flags, section emit/elision. Plus: legacy `## Tasks` files still parse.

**Files.** `src/core/project.ts`, `src/core/tasks.ts`, `src/commands/tasks.ts`, `src/cli.ts`, `tests/project.test.ts`, `tests/tasks.test.ts`, `tests/cli.e2e.test.ts`, `docs/spec.md`, `docs/cli.md`, `example/`.

---

### Step 3 — Contexts

**Scope.** Task lines may carry zero or more `@<string>` context tokens (where `<string>` is any non-whitespace sequence). Item gains `contexts: string[]`. Parser strips trailing `@`-tokens that aren't dates; serializer emits contexts alphabetically before the due date.

CLI:
- `td tasks add --context <name>` (repeatable).
- `td tasks edit <ref> --context <name>` (repeatable; full replacement). `--context ""` clears.

**Tests.**
- Integration: parse contexts; alphabetical emit; preservation through round-trip; `:` not special (`@agenda:isa`, `@agenda:` round-trip).
- e2e: add with multiple `--context`; edit replaces wholesale; `--context ""` clears; mixed contexts + due on same task.

**Acceptance.** Spec bullets on `--context`, alphabetical ordering, opaque `@<string>`, `agenda:` round-trip.

**Files.** `src/core/project.ts`, `src/core/tasks.ts`, `src/commands/tasks.ts`, `src/views/task.ts`, `tests/*`, `docs/spec.md`, `docs/cli.md`.

---

### Step 4 — Preamble notes

**Scope.** Project notes move from `## Notes` to the **preamble** (markdown between frontmatter and `## Available`). `\#` escape on write, unescape on read. Existing hand-written `## Notes` sections are preserved as tail (not surfaced as `project.notes`). `td projects add/edit --notes` writes to preamble.

**Tests.**
- Integration: `parseProject` reads notes from preamble; round-trips; escapes `\#` on write; unescapes on read; `## Notes` in tail is preserved verbatim.
- e2e: `td projects add --notes "..."` puts text above `## Available`; `td projects show` renders it; `td projects edit --notes ""` clears it; `--notes "## Available"` round-trips as literal (escaped).

**Acceptance.** "preamble `--notes \"## Available\"` round-trips as literal", "Existing `## Notes` sections in hand-edited files are preserved verbatim as tail."

**Files.** `src/core/project.ts`, `src/commands/projects.ts`, `src/views/project.ts`, `tests/*`, `docs/spec.md`, `docs/cli.md`.

---

### Step 5 — `## Completed` section + complete/uncomplete behavior

**Scope.** Optional `## Completed` archive section at the bottom, with `YYYY-MM-DD:` date labels (plain text, not markdown headings). `td tasks complete <ref>` moves the item out of its lane into `## Completed` under today's date label. `td tasks uncomplete <ref>` moves it back to `## Available` and clears `completedAt`. Empty date labels removed; empty `## Completed` removed.

Refs become flat across all four sections (Available + Waiting + Deferred + Completed). Completed items keep refs (for `uncomplete`) but are excluded from listings.

**Tests.**
- Integration: parse `## Completed` with multiple date labels; round-trip; complete moves item; uncomplete moves back; empty-date / empty-section cleanup; `completedAt` derived from enclosing label; refs flat across all four sections.
- e2e: `td tasks complete` moves from each lane; same-day completes group under one label; multi-day completes order newest-first; `td tasks list` excludes completed; `td projects show` excludes completed; `td tasks uncomplete` reverses.

**Acceptance.** All spec bullets re: complete/uncomplete and `## Completed`.

**Files.** `src/core/project.ts`, `src/core/tasks.ts`, `src/commands/tasks.ts`, `tests/*`, `docs/spec.md`, `docs/cli.md`.

---

### Step 6 — `td list` cross-project dashboard

**Scope.** New top-level `td list` command. Default output = Tasks (available, context-grouped) + Waiting + Projects. `--all` adds Deferred. Reuses task and project views. Each section has a bold heading.

**Tests.**
- e2e: `td list` against multi-project vault renders all three sections; `--all` adds Deferred; completed items excluded; empty vault prints empty (non-TTY) / hint (TTY); each item shows full ref.

**Acceptance.** "`td list` renders Tasks + Waiting + Projects sections with bold headings; `--all` adds a Deferred section. Completed items are excluded."

**Files.** `src/commands/list.ts` (new), `src/cli.ts`, `src/views/list.ts` (new) or extension to existing views, `tests/cli.e2e.test.ts`, `docs/spec.md`, `docs/cli.md`.

---

### Step 7 — Rendering polish

**Scope.**
- Available lane in grouped views (default `td tasks list`, `td projects show`, `td list`): context-less items first, then for each distinct context a green non-bold `@context` heading + the items under it.
- Bold `Waiting:` / `Deferred:` group headings (distinct from non-bold green context headings).
- Single-lane views (`tasks list --waiting`, etc.) render flat.

**Tests.**
- Integration / e2e: rendering produces the right group order; context-less first; alphabetical-by-first-appearance context order; bold lane headings; flat for single-lane.

**Acceptance.** "Available lane in grouped output renders context-less items first…", "Waiting and Deferred render as flat lists with bold `Waiting:` / `Deferred:` headings."

**Files.** `src/views/task.ts`, `src/views/project.ts`, `src/views/list.ts`, `tests/*`.

---

## Per-step ritual (TDD)

For each step:

1. **Write tests first.** Pick the right level: e2e for any user-touchable behavior (CLI commands), integration for core functions (parse/serialize/lane logic), unit only for branchy isolated logic.
2. **Run the suite.** New tests should fail; existing tests should still pass.
3. **Implement** until new tests pass and existing tests remain green.
4. **Update docs** (`spec.md`, `cli.md`) inline. Spec is the source of truth — keep it consistent.
5. **Run the full suite again.** Lint / typecheck. Zero regressions.
6. **Review against acceptance** bullets in `task-status-and-notes-preamble.md`. Tick off the ones this step covers.
7. **Commit** with a focused message naming the step.

## Out of scope for v0.2

Confirmed in the spec doc — repeating here so we don't accidentally pick them up:

- Subheadings inside lane sections.
- `--context` filter on list (contexts are display-only).
- Auto-migration of legacy `@YYYY-MM-DD` due dates on disk write (lazy on edit only — files re-emit with `!`).
- Auto-migration of `## Notes` content into preamble.
- Sorting by due date within lanes/groups.
- Multi-context display in Waiting/Deferred.
- Special meaning for `:` in contexts.
